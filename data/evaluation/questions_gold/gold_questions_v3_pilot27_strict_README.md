# gold_questions_v3_pilot27_strict.jsonl

Строгая редакция pilot benchmark для оценки RAG-системы по корпусу из трёх книг по ароматерапии.

Файл предназначен для:
- ручной проверки качества benchmark-набора;
- автоматической и полуавтоматической оценки ответов RAG-системы;
- сравнения подходов Naive RAG / Graph RAG / Hybrid RAG;
- перехода к более строгой multi-reference оценке.

---

## Назначение файла

`gold_questions_v3_pilot27_strict.jsonl` содержит эталонные вопросы и ответы для проверки RAG-системы на корпусе из трёх PDF-книг.

Эта версия является более строгой по сравнению с предыдущими редакциями:
- спорные и неоднозначные формулировки сокращены или заменены;
- усилены кейсы с альтернативными валидными источниками;
- ответы приведены к более канонической форме;
- структура записи унифицирована для удобства машинной проверки;
- refusal-кейсы зафиксированы в нормализованной форме.

Ожидаемая каноническая форма отказа:
`В источниках нет ответа.`

---

## Состав корпуса

Benchmark построен по трём книгам:

1. **Даниэль Фести. Энциклопедия ароматерапии для здоровья всей семьи, 2017**
2. **Дэвид К. Хилл. doTERRA. Руководство по химии эфирных масел**
3. **Н.Н. Николаевский. Ароматерапия справочник, 2000**

---

## Баланс набора

Всего в наборе:
- **27 вопросов**

Распределение:
- **9 вопросов на каждую книгу**
- в каждой книге:
  - **3 вопроса из начала**
  - **3 вопроса из середины**
  - **3 вопроса из конца**

Такой баланс нужен, чтобы:
- не смещать benchmark только в начало книги;
- проверять retrieval и generation на разных частях корпуса;
- включать как локальные, так и распределённые по книге случаи.

---

## Распределение по coverage_type

В наборе используются следующие типы покрытия:

- `all_books` — **3**
- `two_books` — **4**
- `one_book` — **17**
- `none` — **3**

Смысл типов:
- `one_book` — корректный ответ поддерживается одной книгой;
- `two_books` — корректный ответ валиден по двум книгам;
- `all_books` — ответ может быть подтверждён всеми книгами корпуса;
- `none` — в источниках ответа нет, и система должна вернуть отказ.

---

## Основная идея strict-редакции

Эта версия набора ориентирована не просто на проверку “похож ли ответ на правду”, а на более строгую оценку:

- есть ли ответ в источниках;
- из правильного ли документа он взят;
- присутствуют ли ключевые смысловые элементы ответа;
- допускаются ли альтернативные, но валидные источники;
- умеет ли система корректно отказывать, когда ответа нет.

То есть benchmark проверяет не только генерацию, но и дисциплину работы с источниками.

---

## Формат одной записи

Каждая строка JSONL содержит один объект со следующими полями:

- `question_id`
- `question`
- `answerable`
- `book_anchor`
- `book_position`
- `coverage_type`
- `knowledge_type`
- `reasoning_type`
- `lexical_type`
- `alt_valid`
- `answer_type`
- `target_items`
- `gold_answer`
- `primary_reference`
- `alternative_references`
- `comment`

---

## Описание полей

### `question_id`
Уникальный идентификатор вопроса.

Пример:
- `festy_beg_001`
- `hill_mid_002`
- `nik_end_003`

Обычно кодирует:
- книгу;
- позицию в книге (`begin`, `middle`, `end`);
- порядковый номер.

---

### `question`
Текст вопроса, который будет подаваться в RAG-систему.

---

### `answerable`
Показывает, должен ли существовать ответ в корпусе.

Значения:
- `true` — ответ в источниках есть;
- `false` — корректный ответ должен быть отказом.

---

### `book_anchor`
Книга, относительно которой вопрос был изначально спроектирован.

Типовые значения:
- `festy`
- `hill`
- `nik`

Это поле полезно для балансировки датасета и анализа покрытия корпуса.

---

### `book_position`
Позиция вопроса внутри книги-источника.

Значения:
- `begin`
- `middle`
- `end`

Это нужно для контроля, что benchmark распределён по всей книге, а не только по первым страницам.

---

### `coverage_type`
Тип покрытия ответа по корпусу.

Значения:
- `one_book`
- `two_books`
- `all_books`
- `none`

Это одно из ключевых полей для интерпретации результата:
- в `one_book` ожидается одно основное подтверждение;
- в `two_books` и `all_books` допускаются альтернативные валидные подтверждения;
- в `none` правильным результатом считается отказ.

---

### `knowledge_type`
Тип знания, которое проверяет вопрос.

Примеры:
- `definition`
- `factoid`
- `list`
- `comparison`
- `cause_effect`
- `procedure`
- `constraint`
- `numeric`

Поле полезно для последующего анализа ошибок по типам вопросов.

---

### `reasoning_type`
Ожидаемая сложность рассуждения.

Примеры:
- `single_hop`
- `multi_source`

Смысл:
- `single_hop` — достаточно одного смыслового фрагмента;
- `multi_source` — возможна проверка ответа через несколько источников или альтернативных формулировок.

---

### `lexical_type`
Тип лексического совпадения между вопросом и источником.

Примеры:
- `exact`
- `paraphrase`
- `synonym`

Это помогает оценивать, насколько retrieval зависит от буквального совпадения слов.

---

### `alt_valid`
Показывает, допустимы ли альтернативные валидные источники.

Значения:
- `true`
- `false`

Если `true`, ответ может быть засчитан не только по `primary_reference`, но и по одному из `alternative_references`.

---

### `answer_type`
Тип ожидаемого ответа.

Примеры:
- `definition`
- `fact`
- `list`
- `number`
- `comparison`
- `refusal`

Это поле удобно использовать в правилах автоматической проверки и в аналитике ошибок.

---

### `target_items`
Минимальные смысловые элементы, которые желательно найти в ответе модели.

Примеры:
- ключевые термины;
- элементы списка;
- числовые значения;
- обязательные части сравнения;
- канонический текст отказа.

`target_items` особенно полезен для:
- быстрой эвристической автопроверки;
- частичного матчинга ответа;
- анализа, какие смысловые части модель пропустила.

Важно:
`target_items` — это не обязательно полный эталонный ответ, а минимальный набор значимых компонентов.

---

### `gold_answer`
Канонический эталонный ответ.

Это основная короткая нормализованная форма ответа, относительно которой удобно:
- сравнивать результат генерации;
- проводить ручную проверку;
- строить эвристики автоматической оценки.

---

### `primary_reference`
Основной эталонный источник ответа.

Структура:
- `source_filename` — имя PDF-файла;
- `source_pages` — список страниц;
- `gold_answer` — формулировка ответа, зафиксированная по данному источнику.

`primary_reference` используется как главный gold-источник при строгой оценке.

---

### `alternative_references`
Список альтернативно валидных источников.

Каждый элемент имеет ту же структуру, что и `primary_reference`:
- `source_filename`
- `source_pages`
- `gold_answer`

Это поле особенно важно для случаев, когда:
- один и тот же ответ подтверждается несколькими книгами;
- допустимы разные, но корректные формулировки;
- benchmark должен поддерживать multi-reference оценивание.

---

### `comment`
Краткое пояснение к вопросу.

Обычно здесь фиксируется:
- почему кейс важен;
- насколько он строгий;
- допускаются ли сокращённые формулировки;
- что именно является условием засчёта;
- является ли кейс refusal, comparison, list-case и т.д.

Это поле особенно полезно при ручной ревизии benchmark.

---

## Как интерпретировать strict-логику

### 1. Если `answerable=true`
Система должна вернуть содержательный ответ, подтверждаемый источником.

### 2. Если `alt_valid=true`
Допускается, что модель опирается не только на основной источник, но и на один из альтернативных.

### 3. Если `coverage_type=none`
Система должна вернуть отказ в канонической форме:

`В источниках нет ответа.`

### 4. Если вопрос является list/comparison/numeric-case
Нужно проверять не только общую тему ответа, но и полноту:
- все ли элементы списка перечислены;
- присутствуют ли обе стороны сравнения;
- указано ли точное число или диапазон.

### 5. Если есть `target_items`
Ответ желательно проверять на наличие этих минимальных смысловых единиц даже в случае перефразирования.

## Чем эта редакция лучше для диплома

`gold_questions_v3_pilot27_strict.jsonl` лучше подходит для дипломного эксперимента, потому что:

- вводит более строгую и единообразную схему;
- уменьшает долю спорных кейсов;
- поддерживает multiple valid references;
- позволяет отдельно анализировать refusal-кейсы;
- подходит для сравнения разных реализаций RAG;
- ближе к реальной задаче проверки grounded-ответов по источникам.

Именно такая структура удобна для следующего этапа:
- запуск Naive RAG;
- запуск Graph RAG;
- запуск Hybrid RAG;
- сравнение по единому benchmark.

## Рекомендуемое использование в evaluation pipeline

Файл можно использовать как вход для:
- `eval_baseline.py`
- последующего анализа run-файлов
- ручной разметки ошибок
- проверки page-level citation quality
- перехода к multi-reference scoring

Практически это означает:
1. прогнать систему по всем 27 вопросам;
2. сохранить ответ, цитаты и найденные страницы;
3. сравнить результат с `gold_answer`;
4. проверить совпадение с `primary_reference` или `alternative_references`;
5. отдельно учитывать refusal-кейсы.

---

## Ограничения набора

Это **pilot benchmark**, а не финальный полноразмерный gold dataset.

Следствия:
- 27 вопросов достаточно для первичной оценки baseline;
- набор хорошо подходит для качественного анализа ошибок;
- для финального сравнения моделей может потребоваться расширение набора;
- часть кейсов всё ещё ориентирована на ручную интерпретацию комментариев.

## Итог

`gold_questions_v3_pilot27_strict.jsonl` — это компактный, но уже достаточно строгий benchmark для оценки RAG-системы по трём книгам корпуса.

Его ключевые особенности:
- баланс по книгам и частям книг;
- поддержка альтернативно валидных источников;
- канонические эталонные ответы;
- нормализованные refusal-кейсы;
- удобство для последующей автоматической и ручной оценки.

Файл можно использовать как основную pilot-версию strict benchmark перед расширением gold-набора.
